# Plan: design-and-implement-task-mx-cadre

## Intent

> "I imagine we add todos as we go — I just say, add this to TODOs and you add it with rich info. The agent is responsible for taking the todoboard and performing all the work. It makes sure the actual board is presented properly and formatted. Then all the long, raw todos are in another section maybe. But basically, I can say pull the next todo item when starting a branch and we can just work from the board."

**Read-back (orchestrator's interpretation, after correction):**
- task-mx-cadre is a **board curator** — it *writes* `.cadre/todos.md`. It does NOT read it on the user's behalf.
- Operations: **intake** (rich-expand new TODOs from user's prompt), **format** (keep board presentation clean), **organize** (logical structure across the board so retrieval-by-orchestrator is trivial).
- The orchestrator (main session) does the pulling — it reads the board, picks the next item, primes the branch. Task-mx makes that pull easy by curating the board well.
- Two-tier representation likely: a clean scannable board + a rich-detail layer for the long raw TODOs.

**Scope confirmed: pure board curation (Q1 = A).** No reads-for-user. No lifecycle/compression yet. No cross-ref maintenance yet. Defer per Last Responsible Moment until felt.

## Could

**Scope** *(Q1, resolved A)*: Pure board curation — intake (rich-expand), format, organize. No retrieval (orchestrator owns that). No lifecycle/cross-ref initially.

**Board shape** *(Q2, resolved A)*: Single file, two sections (clean board + rich detail).

**Refinement from researcher (Q2-bis):**
- *Detail as source of truth; Index is derived.* Mutation = edit Detail; Index regenerates. Eliminates two-tier drift. (via researcher — confidence high, Obsidian/Dataview-collapsed pattern.)

**Operational model** *(Q3, resolved B2)*:
- B2. Hook writes flag → surface hook → orchestrator dispatches subagent (handoff-mx pattern). Battle-tested. Avoids B1's headless-spawn brittleness (cold session per spawn, Windows detach issues, infinite-loop risk per claude-code-guide verification).
- Loop guard: agent's write needs to NOT re-trigger the hook. Mechanism TBD at impl (lock file or path discrimination).

**Calibration correction (mid-could):** initial researcher's "Now/Next/Later — over-engineered urgency scoring" framing was wrong-axis. Cost model is *human cognitive load*, but the maintainer is an LLM — design+compute is the real cost. User's directive: "nothing is overkill." Updated scoring direction: **Taskwarrior-style urgency polynomial** is the load-bearing pick, with LLM-extracts-axes-from-natural-language as the wedge nobody else has.

**Mutation discipline** *(Q4, resolved A)*: Orchestrator drafts proper rich entry with all metadata (Taskwarrior-style fields); task-mx audits, scores, integrates, regenerates index, atomic write.

**Scoring system (via auto_todo_board_research.md):**
- Taskwarrior polynomial (default coefficients): `urgency = w_due·due + w_priority·pri + w_blocking·#blocking + w_scheduled·sched + w_active·active + w_age·age + w_tags·#tags + w_project·proj − w_waiting·wait − w_blocked·blocked`
- DAG primitives: topological sort (cycle detection on add), critical-path boost, ready-set filter
- Aging: linear coefficient, prevents starvation in long-tail
- LLM-assist intake: orchestrator parses natural-language brain-dump into structured fields when drafting the rich entry

**Display shape (index regenerated by task-mx on every fire):**
- Three columns: **Ready** (unblocked, sorted desc by urgency), **Blocked** (sorted separately with blocker callouts), **In Progress** (WIP-limited)
- Term breakdown shown on demand (transparent scoring)
- Top of Ready = pull target

**Schema for rich-detail entries (Taskwarrior-aligned):**
- id (ascending, never reused), title, status (TODO/DOING/DONE/DEFERRED/SUPERSEDED)
- priority (H/M/L), due, scheduled, blockers (list of TODO #s), blocking (derived), tags, project
- impact, effort (RICE-style numerator/denominator — optional)
- created, updated, closed
- detail prose (free-form; conversation transcripts, design notes)

**Schema representation** *(Q6, resolved D)*: YAML frontmatter for hard fields per entry + prose body for narrative. Schema parseable for scoring/sorting; rich content stays prose.

**Migration** *(Q8, resolved A)*: Big-bang. task-mx fires once on ship. Inspects all 31 entries, adds YAML frontmatter inferred from existing prose, archives the 5 DONE to `.cadre/task-mx/archive/<YYYY-MM>.md`. Ambiguous fields marked `?` for user revision. Existing IDs preserved (1–31; new entries continue from 32). Doubles as smoke test.

**Aging policy** *(Q7, resolved E3)*: Single archive on DONE. DONE entries move to `.cadre/task-mx/archive/<YYYY-MM>.md` (or `.cadre/task-mx/archive/<YYYY-MM>.md` per handoff-mx archive-by-month pattern) immediately on close. `todos.md` stays clean — only TODO/DOING/DEFERRED/SUPERSEDED in Detail. git-blame split for DONE acceptable.

## Should

Direction narrowed across all 9 could-questions, with staff-engineer optimizations folded in:

- **Scope**: pure board curation (intake / format / organize). Orchestrator owns retrieval; reads `.cadre/todos.md` directly.
- **Board shape**: single file `.cadre/todos.md` with two sections — `# Index` (regenerated each run by `score.ts`) + `# Detail` (per-entry YAML frontmatter + prose body).
- **Operational model**: B2 + quiescence debounce. PostToolUse hook on writes to `.cadre/task-mx/inbox/*` writes a 0-byte `pending.flag` with debounce ts → UserPromptSubmit surface hook on next user turn (after `now − last_mutation_ts ≥ ~3s`) injects "task-mx pending" → orchestrator dispatches `task-mx-cadre` via Agent tool. Bursts coalesce into one dispatch.
- **Mutation discipline (single-writer)**: Orchestrator writes ONLY to `.cadre/task-mx/inbox/<ts>-<slug>.md` shards (full rich entry — front-matter + prose body). Agent is the SOLE writer of `.cadre/todos.md`. **Loop-guard implementation:** CC's PostToolUse matcher is a tool-name pipe-delimited string, NOT a file-path glob — path discrimination must be implemented as an early-exit guard *inside* the hook body (`if (!toolInput.file_path?.startsWith('.cadre/task-mx/inbox/')) process.exit(0)`), not in the `settings.json` matcher.
- **Schema** (per entry, YAML front-matter + prose body): `id`, `title`, `status`, `priority`, `due`, `scheduled`, `blockers`, `blocking` (derived), `tags`, `project`, `impact`, `effort`, `created`, `updated`, `closed`. Body = free-form markdown.
- **Scoring**: Taskwarrior polynomial with default coefficients, stored in `.cadre/task-mx/weights.json` (config-as-data; agent never computes from memory). Implemented in `.claude/agents/task-mx-cadre.score.ts` (deterministic Bun script) — polynomial + Kahn topological sort with cycle detection + critical-path DP boost + Index markdown render. Mirrors `handoff-mx-cadre.parse.ts`.
- **LLM cleavage**: Agent does only (a) NL → front-matter extraction (when shard has gaps), (b) rich-expansion of sketch entries, (c) merge conflict resolution. Everything deterministic → script.
- **Cycle detection**: refuse-and-emit. Cycle on integration → shard moved to `.cadre/task-mx/inbox/.rejected/<ts>-<slug>.md` with rejection header. `todos.md` invariant preserved.
- **Reflexion self-critique**: pre-rename, agent re-reads against 4-rule watchlist (front-matter completeness / blocker-graph plausibility / schema drift / duplicate-by-title). Single revise-once.
- **Index render**: three top-level sections — `## Ready` / `## Blocked` / `## In Progress` — each a markdown table. Column conventions chosen for unambiguous orchestrator parse:
  - `## Ready`: `#`, title, urgency, **`blocks`** (outgoing edges only; Ready items are by definition unblocked), tags.
  - `## Blocked`: `#`, title, urgency, **`blocked-by`** (incoming edges only), tags.
  - `## In Progress`: `#`, title, urgency, `blocks`, tags.
  - Top of `## Ready` = canonical pull-next (orchestrator reads, no dispatch needed).
  - Term-breakdown stored as HTML comment per Detail entry (`<!-- urgency: due=12*0.8=9.6; blocking=8*2=16; ... -->`) — invisible in CC's markdown renderer, machine-parseable from source. `<details>` HTML element rendering in CC is unverified; comment fallback is zero-render-risk.
- **Aging**: single archive on DONE. DONE entries leave `todos.md` immediately, move to `.cadre/task-mx/archive/<YYYY-MM>.md` (or `.cadre/task-mx/archive/<YYYY-MM>.md`). Detail keeps TODO/DOING/DEFERRED/SUPERSEDED only.
- **Migration**: big-bang on day 1. task-mx fires once, infers schema for all 31 entries, archives 5 DONE, marks ambiguous fields with `?`. Existing IDs preserved (1–31; new from 32).
- **Lifecycle transitions**: orchestrator-driven. Natural-language signal → orchestrator edits `status` frontmatter in shard or directly in Detail entry → task-mx audits.
- **I/O contract**: reads `.cadre/task-mx/inbox/*.md`, `.cadre/todos.md`, `.cadre/task-mx/weights.json`, `.claude/agents/task-mx-cadre.refs.md`. Writes `.cadre/todos.md` (atomic via `.tmp` swap), `.cadre/task-mx/inbox/.rejected/*`. Deletes processing shards. Tools: Read, Write, Bash (for `score.ts`).
- **Three-hook ecosystem**: trigger (PostToolUse on inbox writes), surface (UserPromptSubmit on flag), prime (SessionStart — TBD whether needed; handoff-mx's prime stages events, task-mx's would inject the active board if anything).

## Action Plan

### Context

This is the second `-mx-` agent (after `handoff-mx-cadre`); it follows the same pattern but with adjustments for ~10× higher mutation rate. Prompted by TODO #3 — "First infrastructural agent, because every other agent's work routes through the task board." User intent: "I imagine we add todos as we go — I just say, add this to TODOs and you add it with rich info... I can say pull the next todo item when starting a branch and we can just work from the board." Outcome: a board that maintains itself, an explainable urgency-score-driven `## Ready` queue, and a clean `pull-next` UX (orchestrator-side O(1) read).

**3. `task-mx-cadre.score.ts` (120 min)**
- Inputs: `.cadre/todos.md` (Detail section), `.cadre/task-mx/inbox/*.md` (pending shards), weights config.
- Steps: parse all front-matter → build entry list → run `urgency = Σ w_i · term_i(entry)` per entry → run Kahn topological sort over `blockers` edges (refuses on cycle, returns the cycle path) → compute critical-path-length DP for ready-set boost → render markdown: `## Ready` / `## Blocked` / `## In Progress` per column conventions in Should section.
- **Empty-inbox correctness:** with no shards present, script must exit 0 and emit valid Index from Detail section alone (covered in verification).
- **Atomic write on Windows:** `renameSync(tmp, dest)` is not guaranteed atomic on Win32 when destination exists. Implementation: `process.platform === 'win32'` branch using two-step rename (`renameSync(dest, dest + '.bak'); renameSync(tmp, dest); unlinkSync(dest + '.bak')`), or fall back to `fs.copyFileSync` + `unlinkSync` if rename fails. POSIX path uses single `renameSync`.
- Output: ranked JSON `{ready: [...], blocked: [...], in_progress: [...], cycles: [...] | null}` + rendered Index markdown string.
- CLI: `bun task-mx-cadre.score.ts [--render-only | --integrate <shard-path> | --migrate-dry-run]`.
- `--migrate-dry-run`: emits inferred frontmatter for all existing entries WITHOUT mutating `todos.md`. User reviews diff before atomic write proceeds.

## Reference: handoff-mx-cadre as architectural precedent (read-only context for planning)

Pulled from Explore brief at planning-time so the conversation is grounded.

**handoff-mx-cadre architecture (battle-tested):**
- Subagent (fire-once); model Sonnet
- Input: pre-staged `events.log.processing-<ts>` + `processing.lock`; current `.cadre/handoff.md`; `CLAUDE.md`; ADR log
- Output: `.cadre/handoff.md` (atomic write); archive to `.cadre/handoffs/<ISO-date>.md`
- Dispatch: orchestrator via Agent tool on SessionStart (prime hook injects `additionalContext` orientation; hook-typed agent dispatch is broken in current CC build per ADR-073)
- Three hooks:
  1. **prime** (SessionStart) — pre-stages events for synthesizer; injects current handoff as L1; emits dispatch instruction
  2. **logger** (UserPromptSubmit / PostToolUse / Stop) — appends structured JSON events to `events.log`; per-tool field extraction; non-blocking
  3. **surface** (UserPromptSubmit, post-synthesis) — flag-gated re-injection of fresh handoff
- Parse script (`handoff-mx-cadre.parse.ts`) — deterministic single-shot extraction; replaces 6 Bash calls
- Crash recovery: lock file outlives processing file; retry-safe
- Four anti-patterns: information loss, summarizing-the-summary, vague active items, tool-call log replay

**Current `.cadre/todos.md` shape:**
- Numbered list (1..31), with status markers (`**Status: DONE (date).**`, `**Status: DEFERRED (date).**`, or absent = open)
- DONE entries preserve original contract verbatim under "Original contract (preserved as history):" — no compression of historical content
- Cross-references inline: TODO ↔ ADR ↔ handoff ↔ plan file ↔ PR
- Inline blockers: "Open dependencies:" bullets
- Approximately: 5 DONE, 2 DEFERRED, 24 open/pending; numbering not strictly chronological (some out-of-order: 26 after 28, 27 after 28)
- No automated maintenance to date — hand-maintained
