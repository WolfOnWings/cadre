# Prior-Art Research: 16 Workflow Rules

**Date:** 2026-04-22
**Scope:** Locate named frameworks, seminal papers, community conventions, and practitioner-cited sources for 16 rules spanning git/review architecture, agent architecture, and human-AI collaboration practices.

---

## Theme 1: Git and Review Architecture

### 1. Git worktrees convention (sibling directory, per-feature, multi-session reuse)

**Prior art:**
- **Sibling directory convention.** The "bare repo + sibling worktrees" layout is a well-established community pattern. Andreas Schneider's "Sliced bread: git-worktree and bare repo" (cryptomilk.org, 2023), Pablo Arias "Boost Your Workflow: Exploring Git Worktrees" (2023), and Nick Nisi's "How I use git worktrees" all describe a structure where a hidden `.bare/` administrative directory lives alongside sibling worktrees named by branch. **Credibility: medium** (practitioner blogs with wide citation).
- **Naming pattern `<project>-<branch>` or `<repo>.worktrees/<branch>`.** Documented in sellout's gist "Git worktree layout" on GitHub and DataCamp's Git Worktree Tutorial; convention aligns with git's own docs warning against nested worktrees. **Credibility: medium.**
- **Per-feature/task scoping with multi-session reuse.** Implied across the practitioner literature (treat a worktree as a durable per-branch workspace), and now standard in AI-assisted coding pipelines — see Ben Vargas's "Bare Git Worktrees AGENTS.md" (2026) and spec-compare/Cameron Sjo's research on worktree use for parallel spec-driven agents. **Credibility: medium.**

**Confidence: High** that the sibling-directory layout and name-by-branch convention is widely adopted; **medium** on the specific "multi-session reuse until merge" discipline being explicitly codified — it is implied rather than formally named.

**Contradictory evidence:** git's own docs are neutral on layout; some tutorials (DataCamp) show nested worktrees under `<repo>/worktrees/`, which the community generally considers anti-pattern. No single canonical source.

### 2. Plan before execute

**Prior art:**
- **Spec-Driven Development (SDD).** GitHub Spec Kit (github/spec-kit, 2025), AWS Kiro, Tessl, and BMAD-METHOD all formalize a `spec → plan → tasks → execution` pipeline. Martin Fowler's "Understanding Spec-Driven Development" (martinfowler.com, 2026) and Thoughtworks' technology radar entry on SDD (2025) treat it as a first-class AI-era engineering practice. **Credibility: high.**
- **Intentional separation of propose/review/execute acts.** Has older roots in waterfall-era design-before-code and the SDLC more generally; Roger Wong ("Spec-Driven Development: It Looks Like Waterfall," 2026) explicitly notes this lineage. **Credibility: medium.**
- **Cursor Plan Mode, Claude Code plan mode, Gemini CLI planning phases** are tool-level implementations. Addy Osmani, "How to write a good spec for AI agents" covers the contemporary practitioner consensus. **Credibility: medium.**
- **Separation of duties (SoD)** from compliance/audit tradition (Kaiburr, GSA CIO IL-22-01 on DevSecOps, 2022) is the formal progenitor of the proposer/reviewer/executor split, though it's about people rather than cognitive acts. **Credibility: high.**

**Confidence: High.** Plan-before-execute under the SDD umbrella is now mainstream; the intentional separation of "propose / review / execute" echoes classical SoD and TDD's red/green/refactor phase separation.

### 3. Three-stage review architecture (commit/push/merge gates; escalating reviewers)

**Prior art:**
- **Quality gates in CI/CD.** Propel's "Setting Up Code Quality Gates," Augment Code's "AI Code Review in CI/CD Pipeline" (2026), and GitLab/GitHub branch protection docs establish the pattern of pre-commit hooks → pre-push checks → PR gates as the standard staged pipeline. **Credibility: high.**
- **Tiered review framework.** Kluster.ai's "Best Practices for Code Review" explicitly names "Tier 1: automated linting/tests, Tier 2: peer review, Tier 3: senior/security" — a near-exact match to the escalating-reviewer pattern. Ideamotive (2026) echoes this. **Credibility: medium.**
- **Gated commits.** Wikipedia's "Gated commit" article documents the CI-merges-validates-commits workflow pioneered by tools like Electric Cloud/ElectricAccelerator. **Credibility: medium.**
- **Pre-commit and pre-push review (AI-era).** Nous Research's hermes-agent `requesting-code-review` skill explicitly separates pre-push LLM review as a distinct gate, returning APPROVE/REQUEST_CHANGES. **Credibility: medium.**
- **SAFe / three-branch dev-staging-prod strategies** give the macro version of staged promotion with escalating scrutiny. **Credibility: high.**

**Confidence: High** on staged review with escalating scrutiny; **medium** on the specific 1/2/N reviewer-count progression — it is implied in tiered-review guidance but not named as a specific formula.

### 4. Coherence-based commit sizing (book metaphor)

**Prior art:**
- **Atomic commits.** Wikipedia "Atomic commit"; Sandro Dzneladze's "A Developer's Guide to Atomic Git Commits" (Medium); Nicolas Jourdan's "Atomic Commits & Conventional Commits" — all define atomic as "one semantic concern, one coherent testable change." The arXiv paper "Detecting Multiple Semantic Concerns in Tangled Code Commits" (2026) gives formal definition: "syntactically coherent and semantically cohesive — addressing exactly one semantic concern." **Credibility: high.**
- **Narrative-history framing.** FutureLearn's "Telling stories with your Git history" and Medium's "gitlogue/GitNarrative" projects explicitly treat commits as "chapters" in a "story." The book/chapter metaphor is a direct practitioner motif. **Credibility: medium.**
- **Size-independent framing ("100-line rename is still one sentence").** Kent Beck's small-batch philosophy via TCR (see #5) and Jake Goulding's "Refactor and make changes in different commits" both argue cohesion, not line count, is the unit of measure. **Credibility: medium.**

**Confidence: High** on semantic-coherence-over-size; **medium** on the exact sentence/page/chapter metaphor — coherent-narrative framing is widespread but the specific three-level ladder appears **partially novel** as a codified rule.

### 5a. Machine-parseable commit messages

**Prior art:** **Conventional Commits specification** (conventionalcommits.org v1.0.0, descended from Angular's convention). Adopted as the de facto standard. Enables semantic-release, auto-changelog, semver-automation. **Credibility: high. Confidence: High.**

### 5b. "When uncertain whether to commit or continue, commit"

**Prior art:**
- **Kent Beck's TCR (test && commit || revert)** (Medium, 2018; InfoQ "How Practicing TCR Reduces Batch Size"). The exact heuristic "when tests pass, commit" is Beck's; the inverse "when in doubt, commit" is a natural corollary that sits inside TCR's ethos of smallest-possible-steps. **Credibility: high.**
- **XP/TDD tradition** (Beck, *Extreme Programming Explained*, 1999) — "smallest step that works" is a foundational value. **Credibility: high.**

**Confidence: High** that the heuristic has deep prior art in Beck's TCR/XP lineage; the specific wording "commit when in doubt" is adjacent to rather than identical to Beck's formulation.

### 5c. Logical-chunk commits (scaffolding/tests/impl/refactor)

**Prior art:**
- **Separate refactoring from feature commits.** Jake Goulding's "Refactor and make changes in different commits" (2012), int13's "Better Commits - Refactorings," CleverThis "Writing High Quality, Well Scoped, Commits." Consensus: refactor-then-feature in separate commits halves review cost. **Credibility: medium.**
- **TDD red/green/refactor** (Kent Beck; Martin Fowler bliki "TestDrivenDevelopment") naturally produces test/impl/refactor commit triples. **Credibility: high.**
- **Conventional Commits types** (feat/fix/refactor/test/chore) codify the taxonomy. **Credibility: high.**

**Confidence: High.**

### 6. Unconditional end-of-session retros

**Prior art:**
- **Sprint/iteration retrospective is mandatory in Scrum** (scrum.org "What is a Sprint Retrospective?"; SAFe "Iteration Retrospective"). Agile Manifesto principle 12: "At regular intervals, the team reflects." **Credibility: high.**
- **Norman Kerth's Prime Directive** (*Project Retrospectives*, 2001): "regardless of what we discover, everyone did the best job they could given what they knew" — the exact mechanism for "tweaks and reframes, not blame." **Credibility: high.**
- **"Never skip retrospectives."** NextAgile and Scrum.org explicitly frame skipping retros as "leaving performance gains on the table." **Credibility: medium.**

**Confidence: High.**

### 7. Base-3 recursive retrospectives / scale-invariant templates

**Prior art:**
- **Hierarchical retrospectives.** LeSS's "Overall Retrospective" (less.works), SAFe's Multi-Team / Inspect-and-Adapt retrospective, Scrum@Scale's Scrum-of-Scrums retrospective. All roll team retros up to program/portfolio retros. **Credibility: high.**
- **Scaled/multi-team retro structures.** ScatterSpoke, scalingpatterns.org "Multi-Team Retrospective," Methods&Tools "Running Scaled Retrospectives in Agile." **Credibility: medium.**
- **Fractal / scale-invariant analysis.** Mathematical foundation in fractal geometry (Mandelbrot) and self-similarity; Wikipedia "Self-similarity" / "Scale invariance." The idea of applying one template at every altitude borrows from fractal thinking and from holonic/holon theory (Arthur Koestler, *The Ghost in the Machine*, 1967). **Credibility: medium.**
- **Base-3 specifically.** Miller's Magical Number 7±2 (Psychological Review, 1956) gives the working-memory floor; a base of 3 is below 7±2 and aligns with the "rule of three" common in rhetoric, programming (Don Roberts / Fowler's Rule of Three for refactoring), and decision-science. **Credibility: medium.**

**Contradictory evidence:** Most scaled-retro frameworks use two-tier (team → program) or three-tier (team → program → portfolio) rollups, not recursive fan-in with a fixed branching factor. The **specific base-3 recursive rollup** with the **same template at every altitude** does not appear to have a direct named prior-art match — it is a **plausibly novel** synthesis.

**Confidence: Medium** on the general idea of hierarchical retrospectives (prior art strong); **low** on the specific base-3 recursive scheme — no direct match found, though component concepts (scale invariance, rule of three, LeSS/SAFe rollups) are well-established.

## Theme 2: Agent Architecture

### 8. Main session as default orchestrator (with recursion limits)

**Prior art:**
- **Anthropic, "How we built our multi-agent research system"** (anthropic.com/engineering, 2025). Lead Researcher as primary agent that spawns subagents; each subagent has fresh context. The exact pattern. **Credibility: high.**
- **Anthropic, "Building Effective Agents"** (Schluntz & Zhang) defines orchestrator-workers as a named pattern. **Credibility: high.**
- **Claude Code subagent architecture** (code.claude.com/docs/en/sub-agents) explicitly treats the main conversation as primary/orchestrator, with subagents spawned on demand. OpenCode's `primary` vs `subagent` mode is the same distinction. **Credibility: high.**
- **Recursion depth limits.** beam.ai "6 Multi-Agent Orchestration Patterns for Production (2026)" documents spawn-depth limits (default 1, max 2) to prevent unbounded nesting; context-window and latency-accumulation arguments motivate the cap. **Credibility: medium.**
- **Arize AI "Orchestrator-Worker Agents: A Practical Comparison"** (2025) covers common frameworks. **Credibility: medium.**

**Confidence: High.** Pattern is well-established, named, and documented at Anthropic itself.

### 9. Agent contract iron law (input schema, output artifact, declared R/W footprint)

**Prior art:**
- **Schema-as-contract for tools/agents.** APXML's "Tool Input and Output Schemas," MCP architecture (modelcontextprotocol.io) — both treat schemas as formal contracts. Sopan Deole's "Data Contracts for Agents" (Medium, 2026) explicitly names "agent data contracts" with explicit I/O typing. **Credibility: medium.**
- **Software engineering lineage.** Bertrand Meyer's Design by Contract (Eiffel, *Object-Oriented Software Construction*, 1988/1997). Command-Query Separation (Wikipedia; Fowler bliki) is the canonical reference for making side effects structurally explicit. **Credibility: high.**
- **Pure functions / explicit side effects.** Haskell's IO monad, Elm's "No runtime exceptions" discipline, and CQS all codify "declared effects = contract." **Credibility: high.**
- **Principle of least privilege** (Saltzer & Schroeder, "The Protection of Information in Computer Systems," 1975) is the foundational text for "declared footprint" in security terms. **Credibility: high.**
- **Addy Osmani, "The Code Agent Orchestra"** treats contract-per-agent as a named architectural primitive. **Credibility: medium.**

**Confidence: High.** The concept is richly grounded; calling it an "iron law" specifically for agent architecture is a local framing but the underlying discipline has deep software-engineering provenance.

### 10. Three-tier project file namespace (human-facing root + hidden op state + hidden harness + product src)

**Prior art:**
- **Dotfiles convention.** Unix/Linux tradition that hidden files start with `.` and hold configuration — ArchWiki "Dotfiles," MIT Missing Semester "Dotfiles," Atlassian "How to Store Dotfiles." **Credibility: high.**
- **Hidden admin directory for tooling.** The `.git/`, `.github/`, `.vscode/`, `.idea/`, `.claude/` pattern — each tool claims its own hidden subdirectory without polluting the human-facing root. Widespread convention; no single canonical source. **Credibility: high.**
- **kriasoft/Folder-Structure-Conventions** on GitHub documents top-level layouts (`src`, `docs`, `test`, `build`, `tools`, plus README/LICENSE at root). **Credibility: medium.**
- **Emergent structure.** No named principle for "contents emerge as needed, no pre-prescribed taxonomy" in repo-layout literature specifically, but connects to YAGNI and Martin Fowler's evolutionary architecture (Ford & Parsons, *Building Evolutionary Architectures*). **Credibility: medium.**

**Contradictory evidence:** Most layout guides prescribe fixed top-level taxonomies rather than emergent ones. The "emerge-as-needed" discipline for the hidden op-state directory is an uncommon stance.

**Confidence: High** on the general three-tier hidden/source/product separation (inherited convention); **medium** on the specific emerge-as-needed stance for op state, which is a YAGNI-adjacent but not directly documented pattern.

## Theme 3: Working With Me (Human-AI Collaboration)

### 11. Dialogue verbosity calibration (0-10 "yanking" scale)

**Prior art:**
- **Explicit verbosity calibration** in LLM systems. Anthropic's prompting guide (platform.claude.com) explicitly discusses calibrating response length. Response-length control is a standard prompt-engineering lever. **Credibility: high.**
- **Register calibration in linguistic pragmatics.** Halliday's register theory (field/tenor/mode; *Language as Social Semiotic*, 1978) is the classical source for dialogue register — though applied to human communication broadly, not a numeric scale. **Credibility: high.**
- **The "0-10 scale"** specifically — similar numeric scales appear in coaching ("rate your confidence 1-10"), Kahneman's WYSIATI/decision-scaling work, and the Likert-scale tradition (Rensis Likert, 1932). No direct match for a verbosity-specific 0-to-10 dial. **Credibility: low.**

**Contradictory evidence:** Most LLM guidance recommends context-sensitive calibration over fixed numeric dials. The numeric dial for dialogue register appears **plausibly novel** as a codified heuristic.

**Confidence: Medium** — calibrating verbosity is universal; the specific 0-10 "yanking" scale with named middle target is **likely novel** local notation.

### 12. Exchange rhythm (many short turns vs. few monolithic turns)

**Prior art:**
- **Ping-Pong Pair Programming.** Anthony Sciamanna (2015), Open Practice Library "Ping-Pong Programming," O'Reilly *The Agile Developer's Handbook* — explicit codification of short-cycle collaborative rhythm. **Credibility: high.**
- **TDD red/green/refactor 1-10 minute cycles** (Kent Beck; Uncle Bob "The Cycles of TDD"). **Credibility: high.**
- **Iterative co-creation / short-cycle design.** Design sprint methodology (Jake Knapp, *Sprint*, 2016), and the broader Agile tenet of short feedback cycles. **Credibility: high.**
- **Multi-turn conversation quality.** PromptHub's "Why LLMs Fail in Multi-Turn Conversations" argues that many short turns are paradoxically harder for LLMs than fewer longer ones — a counterpoint. **Credibility: medium.**

**Confidence: High** on the general rhythm preference; the framing of "many-short-turn" as a **dialogue pattern** with a human collaborator is well-established via ping-pong pairing and design sprints.

### 13. Refuse to pre-invent taxonomy / structure emerges

**Prior art:**
- **YAGNI** ("You Ain't Gonna Need It") — Kent Beck, XP (late 1990s); Martin Fowler bliki. The direct named principle. **Credibility: high.**
- **Premature abstraction / speculative generality.** Martin Fowler's *Refactoring* code smells taxonomy includes "Speculative Generality"; codeworld.blog "Premature Abstraction in Software Design"; Jason Swett "Premature Generalization." **Credibility: high.**
- **Rule of Three** — Don Roberts via Martin Fowler's *Refactoring* (1999). Wait for three examples before abstracting. **Credibility: high.**
- **Emergent design / evolutionary architecture.** Ford & Parsons *Building Evolutionary Architectures*; Agile Manifesto principle 11 ("best architectures emerge from self-organizing teams"). **Credibility: high.**
- **Chesterton's Fence** (G.K. Chesterton, 1929) — inverse framing: don't remove structure without understanding; complement to "don't invent structure without need." **Credibility: high.**
- **"We don't know yet" as valid answer.** Connects to John Keats's Negative Capability (letter to brothers, 1817) — "capable of being in uncertainties, mysteries, doubts, without any irritable reaching after fact & reason." **Credibility: high (as a classical concept).**

**Confidence: High.** Deeply grounded across multiple traditions.

### 14. Corrections at earned scope (don't overcorrect to blanket rule)

**Prior art:**
- **Overgeneralization.** Psychology Tools "Overgeneralization," cognitive distortion literature (Aaron Beck's cognitive therapy, 1970s). **Credibility: high.**
- **Generalization Bias in Science.** Peters et al., "Generalization Bias in Science" (*Cognitive Science*, 2022) documents the systematic tendency to overgeneralize from narrow findings. **Credibility: high (peer-reviewed).**
- **Overcorrection psychology.** Neurolaunch "Overcorrection Psychology" documents overcorrection as a discrete failure mode — driven by desire to avoid past mistakes, overriding proportional response. **Credibility: medium.**
- **Corrective feedback scope.** Applied linguistics literature (Ellis; Bitchener & Ferris) distinguishes "focused" vs "unfocused" WCF — narrow correction often outperforms broad because broad overloads processing. Applies to human learners; transfers directly. **Credibility: high.**
- **Karl Popper's falsification / fallibilism** — corrections should be as specific as the error, not broader (*The Logic of Scientific Discovery*, 1934). **Credibility: high.**

**Confidence: High.**

### 15. Defer and TBD as achievements (productive ambiguity)

**Prior art:**
- **Last Responsible Moment (LRM).** Core principle of Lean Software Development (Mary & Tom Poppendieck, *Lean Software Development*, 2003). "Delaying commitment until the cost of not making the decision exceeds the cost of making it." **Credibility: high.**
- **Defer Commitment.** Explicit Lean principle documented across PMI Agile Certified Practitioner materials, O'Reilly *Being Agile*. **Credibility: high.**
- **Set-Based Concurrent Engineering** (Toyota product development; Allen Ward; Durward Sobek) — explicitly keeping multiple alternatives alive as a *deliverable*, not a failure. **Credibility: high.**
- **Integrative Thinking** (Roger Martin, *The Opposable Mind*, 2007) — holding conflicting ideas simultaneously as raw material for superior synthesis. **Credibility: high.**
- **Negative Capability** (Keats, 1817) — tolerating uncertainty as a cognitive virtue. **Credibility: high (classical).**
- **Caveats.** Ben Morris "Last Responsible Moment should address uncertainty, not justify procrastination" (ben-morris.com) notes the failure mode of weaponizing LRM as avoidance. **Credibility: medium.**

**Confidence: High.**

### 16. Creative drilling with externalized tab stack (depth-first ideation with branch tracking)

**Prior art:**
- **Parking Lot technique** (Silicon Valley Project Management; NN/Group "Parking Lots in UX Meetings and Workshops"). Classic meeting-facilitation practice: capture off-topic items for later, return before meeting end. Direct match for "externalized stack of open branches." **Credibility: high.**
- **Depth-first search with stack** — classical CS (CLRS; Russell & Norvig *AIMA*). The algorithmic substrate. **Credibility: high.**
- **Tree of Thoughts (Yao et al., NeurIPS 2023)** — explicit LLM reasoning via branching tree with backtracking. Direct formal analogue for conversational branch walking. **Credibility: high (peer-reviewed).**
- **Yak Shaving** (coined by Carlin Vieri, MIT AI Lab, 1990s) — names the depth-first tangent pattern. Lobsters "What's your current yak-shaving depth?" explicitly tracks depth. **Credibility: medium.**
- **Mind mapping / externalized working memory** (Tony Buzan, *Use Your Head*, 1974). Externalizing cognitive structure to offload working memory. **Credibility: medium.**
- **Miller's 7±2** (Psychological Review, 1956) — motivates externalization: once branch stack exceeds working memory, external tracking is necessary. **Credibility: high.**
- **Conversation branching in LLM systems** — Haystack "Good Listener: Memory in Conversational Agents"; AWS AgentCore Memory. Branching conversations as a first-class feature. **Credibility: medium.**

**Confidence: High** on component prior art — every ingredient (DFS, stack, parking lot, mind mapping, Miller's limit, yak shaving, ToT) is well-established. The **specific synthesis** of "assistant actively maintains the stack of open parent branches during drilling and resurfaces them on resolution" as a codified AI-conversation contract appears to be a **plausibly novel** local articulation — no direct prior-art source names this as a conversational-AI discipline, though it is obvious-in-hindsight once the components are named.

## Knowledge Gaps

1. **Exact "book metaphor" commit sizing ladder (sentence/page/chapter).** Narrative-history framing is widespread; the specific three-level ladder mapping to commit/push/merge is likely local.
2. **Base-3 recursive retrospective with same template at every altitude.** Hierarchical retros exist (LeSS/SAFe/Scrum@Scale); scale-invariant analysis exists (fractals); base-3 rollup as an explicit rule — no direct prior-art match found.
3. **0-10 "yanking" verbosity scale.** Numeric verbosity dials are rare in the LLM literature; most guidance is context-sensitive rather than scalar.
4. **Conversational-AI stack-walking contract (rule 16).** Components are grounded; the specific AI-dialogue framing appears local.
5. **"Emerge-as-needed" top-level directory policy.** YAGNI-adjacent but not directly documented as a repo-layout rule.

## Cross-Cutting Observations

- **The rules cluster around a coherent intellectual lineage:** Lean (Poppendieck), XP/TDD (Beck), Agile Manifesto (2001), Design by Contract (Meyer), evolutionary/emergent design (Fowler, Ford & Parsons). Several rules (5b, 5c, 12, 13, 15) are natural extensions of this tradition into the human-AI collaboration domain.
- **A second cluster draws on AI/agent architecture literature (2023-2026):** Anthropic's building-effective-agents canon, SDD tools (Kiro/Spec-Kit), ToT prompting, multi-agent research systems. Rules 2, 8, 9 are at home there.
- **A third cluster is cognitive-science grounded:** Miller 7±2, overgeneralization, Negative Capability, Integrative Thinking. Rules 11, 14, 15, 16 draw from here.
- **Genuinely novel (no single prior-art match) rules:** 4 (book-metaphor ladder), 7 (base-3 recursive retros), 11 (numeric verbosity dial), and 16 (assistant-maintained branch stack) are syntheses where the component ideas are grounded but the specific codification is local. These are the highest-value places to either cite as "our synthesis of X, Y, Z" or develop further as named local contributions.
- **Strongest prior art:** rules 2, 5a, 6, 8, 9, 13, 15 — each has multiple independent, high-credibility sources.
- **Weakest prior art:** rule 11 (numeric verbosity dial) — register calibration is old, but the specific scalar framing lacks direct precedent.
