# HANDOFF

Append-only session log. Each entry = one session. Format:

- **Narrative** — prose walkthrough so a fresh instance can reconstruct what happened and why.
- **Decisions** — condensed bulleted list of architectural/tactical decisions made this session. (In the future, these get mirrored into a master historical ADR log; handoff carries the recent slice, ADR log carries the full history.)
- **Open threads / next-up** — what's outstanding and what the next session should expect.
- **Pointers** — where the rich context lives if the fresh instance needs to dig deeper.

## 2026-04-22 — Bootstrap session: scaffold the repo, settle the architecture

### Narrative

This was the first session on the project. The repo started as a bare directory (`C:\Users\duham\Projects\kata_v2\`) containing nothing but `CLAUDE.md`. By the end of the session it was a live public GitHub repo at `https://github.com/WolfOnWings/cadre`, with a single root commit, partial branch protection in place, a 24-item task board under `.cadre/todos.md`, and a decomposed architectural doctrine distributed across CLAUDE.md, the task board, and the bootstrap plan file. The project was also renamed mid-session from *Kata* to *Cadre*.

The user kicked off with: "this is kata v2. V1 was becoming too monolithic and bloated. First we need to get some basic things set up." They wanted to work through a concrete list of git-setup practices and decide each one individually. I offered four possible starting axes (repo hygiene, Kata's architecture bones, intent document, v1 salvage). They picked repo hygiene.

**Walkthrough mode.** The user had a 52-item checklist (their own curated list of git-hygiene + session-workflow + recovery practices, annotated with 🟦 defensive / 🔺 offensive / ● neutral markers). We walked it one item at a time via AskUserQuestion. For each, I offered a short usefulness statement plus a recommendation; the user decided include / skip / defer. This single-decision cadence paid off — it forced real thought on every item and surfaced architectural decisions the walkthrough itself didn't anticipate.

**Mid-walkthrough rename.** The project was codenamed *Kata* in v1 (which had drifted into bloat). Partway through, the user paused the walkthrough to rename. I offered four candidates (Loom, Lodestone, Cadre, Atlas) based on the CLAUDE.md doctrine. They picked **Cadre** — small, disciplined unit, commander orients by intent. Fits the Mission Command doctrine that CLAUDE.md already enshrines.

**Emergent artifacts (not in the original walkthrough).** The walkthrough exposed a need for two first-class root artifacts the user hadn't originally planned for: `TODOS.md` (forward-looking task board) and `HANDOFF.md` (session log — this file). Both get agents eventually: a taskboard runner for TODOS and a handoff maintainer for HANDOFF. TODOS.md was introduced when the user realized deferred items needed to land somewhere besides scrollback. HANDOFF.md as a session-level log (not per-commit) was rescoped after I conflated per-commit narrative (belongs in commit messages) with per-session narrative.

**Three-review architecture.** A central architectural commitment emerged from the in-session discussion: a three-stage review system anchored to git events. Commit review at pre-commit hook (sentence scope, self-audit), push review at pre-push hook (page scope, self + one stranger agent), merge review at PR gate (chapter scope, stranger swarm → main Claude orchestrates synthesis → human decides). The *book metaphor* (sentence / page / chapter) replaces line-count as the sizing rubric — coherence, not length, decides. A 100-line rename is still one sentence. Gates are unconditional (they fire on every git operation), and each review brief is delivered just-in-time into the acting agent's context. The hot/cold review vocabulary we briefly developed was retired in favor of the commit/push/merge names (anchored to the git verb rather than the thermal metaphor).

**Iron law of agents.** Structural requirement, not a convention: every agent in Cadre has a declared (input schema) → (output artifact) contract plus a declared file footprint. Applied retroactively to every agent named or implied in the session (taskboard runner, handoff maintainer, ADR/wiki fetcher, push-review stranger, merge-review reviewers, risk detector, etc.). This is the v1 failure-mode antidote: agents without artifact structure drift into undocumented side effects.

**File hygiene three-tier namespace.** Root holds exactly the human-facing files (CLAUDE.md + the private artifacts); `.cadre/` is Cadre's infrastructure namespace (contents deliberately unspecified — *contents emerge*, no prescribed taxonomy); product source in its own namespace. This discipline is explicitly load-bearing, not cosmetic, because prior Kata attempts blew up partly on file-hygiene decay.

**The meta-conversation.** Late in the walkthrough the user flagged that I kept "yanking us around corners with taxonomies and early bounding" — colonizing their idea space with pre-structured recommendations. We diagnosed the pattern (weak proprioception of level + training pressure toward helpful completeness + heuristic substitution) and traced it to a need for a preresponse-injected reminder system. The user then recalled they had built exactly this in v2 — **engrams**, YAML objects with `rule:` / `rationale:` / `examples:` meant to bridge memories and hooks, but whose implementation failed on semantic-hash fuzzy matching. The fix: revive the concept with unconditional firing instead of semantic matching. This became TODO #15, with the full conversation transcript preserved in Appendix A of the plan file (and duplicated into the TODO itself) so the motivation doesn't decay. This session is itself Exhibit A for why the engram mechanism matters.

**Plan approval.** After walking all 52 items, I wrote a 300+ line plan file at `~/.claude/plans/let-s-start-with-1-peppy-kernighan.md` capturing every decision, the 23 seeded TODOs, the CLAUDE.md seed content for future encoding, and the verification checks. The user walked through the plan section-by-section, flagged three issues (stale line, repetition, over-long TODO #15 block), and pushed a major restructure: architectural doctrine should ride with the TODO that owns it, not accumulate at the top of the plan. Sections 4 (three review stages), 7 (frequency model), 9 (merge review artifact flow) moved into TODO #14. Sections 8 (iron law) and 10 (file hygiene) moved into the CLAUDE.md seed content for TODO #1. Plan shrunk substantially. Plan was approved.

### Decisions this session (condensed — for eventual ADR extraction)

Numbered for session-level reference. Format: one-line decision.

1. **Rename Kata → Cadre.** Small disciplined unit under a commander's intent. Fits Mission Command doctrine.
2. **Main Claude session is the orchestrator.** Specialized agents are justified by scope, not by role. No separate orchestrator agent — recursion stops buying anything at that depth.
3. **Iron law: every agent has (input schema) → (output artifact) with declared file footprint.** Structural requirement, not a convention.
4. **Three-review architecture.** Commit review (pre-commit, sentence scope, self-audit) / push review (pre-push, page scope, self + stranger) / merge review (PR gate, chapter scope, stranger swarm + synthesis → human).
5. **Book metaphor for sizing.** Commit = sentence, push = page, merge = chapter. Coherence, not length, decides. No line-count gates.
6. **Gates are unconditional.** Agents attempt the git operation; the gate decides pass/fail. Agents don't decide whether to invoke a gate.
7. **Review briefs delivered just-in-time.** Injected by each hook into the acting agent's context at the moment of the gate. Not preloaded.
8. **Self-audit prompt is inverted.** "What would make me refuse to merge this?" instead of "what did I get wrong?" — counteracts self-preference bias.
9. **Hot/cold review vocabulary retired.** Replaced by commit/push/merge-review naming (anchored to git verb, not thermal metaphor).
10. **No auto-merge.** Every merge is human-decided. Merge orchestrator produces a 10–20-line human dashboard.
11. **Consensus/vote mechanism deferred** pending the user's agent-voting paper.
12. **File hygiene three-tier namespace.** Root = human-facing files (CLAUDE.md + gitignored privates); `.cadre/` = infrastructure (contents emerge, no prescribed taxonomy); product source in own namespace.
13. **`.cadre/` contents emerge — no prescribed subfolder taxonomy.** Created at moment of need, justified by what's in them.
14. **TODOS.md and HANDOFF.md live at `.cadre/todos.md` and `.cadre/handoff.md`, gitignored.** `.cadre/` as a whole is gitignored for now; pattern narrows if some `.cadre/` content ever needs to be shared.
15. **Repo is public** at github.com/WolfOnWings/cadre (GitHub Free tier doesn't allow branch protection on private repos).
16. **Partial branch protection on main.** Require PR (zero approvals), block force-push, block deletions, enforce admins. Require-CI-green deferred to TODO #23.
17. **Merge queue deferred.** Blocked on required status checks → blocked on CI → deferred.
18. **Plan-before-execute is the workflow rule.** Non-trivial sessions start in plan mode; approval gates execution.
19. **Conventional Commits enforced by commit-msg hook** (deferred to TODO #21 install; manually followed in the root commit).
20. **When unsure whether to commit or keep going, commit.** Self-enforcing heuristic.
21. **Worktrees: per feature/task, outside the repo.** Location: `C:\Users\duham\Projects\cadre.worktrees\<branch>`. Same worktree reused across sessions until the branch merges.
22. **`git worktree prune` automated at session-end.**
23. **Retros end-of-session, non-failure-gated.** Goal: find tweaks/reframes for Cadre, not blanket rules.
24. **Defer and TBD are first-class achievements.** The level of resolution matters as much as resolution.
25. **Co-Authored-By trailer is harness default.** No action needed; Claude Code injects automatically.
26. **Recovery-playbook git commands are orchestrator-operated.** User doesn't need muscle memory for commands they won't type. Three recovery commands deserve named Cadre wrappers: `cadre bisect`, `cadre recover`, plus a destructive-git safety hook.
27. **Pre-commit framework install deferred** to TODO #21 (design review required).
28. **CI workflow deferred** to TODO #22 (design review required).
29. **ADR template deferred** — create `decisions/` + `0000-template.md` only when the first real ADR is being written. Parsimony.
30. **Pattern library** (versioned library for reusable primitives — agents, skills, prompts, hooks) deferred to TODO #17.
31. **Recursive retro architecture (base-3).** One scale-invariant template applied at every altitude. L0 = handoffs + ADR log; L1 consumes 3 handoffs; L2 consumes 3 L1s; Ln consumes 3 L(n-1)s. Abstraction carried by input compression, not altitude-specific prompting. CLAUDE.md at every level as alignment anchor. Citation chain from L(n) back to raw handoffs. Design deferred to TODO #25.
32. **Iterative-creation collaborative pattern.** Complex artifacts are built in layers (L0 skeleton → L1 prose → L2 cited/grounded → L3+ refinement). The load-bearing discipline is **skeleton-before-content** — a full outline exists before any part gets deep. Lockstep advancement across parts is one mode; depth-first on a productive part is also fine once the skeleton is in place. Judgment, not law. Prevents the premature-bounding failure mode (see #15 engram context): at each layer, the user can steer shape before content ossifies. Named during the CLAUDE.md workflow-section drafting. Candidate pattern-library entry when TODO #17 exists.
33. **Dispatch subagents clean (orchestrator rule).** Subagents start context-blank; the prompt is their total world, so named sources or framing hints seed confirmation bias. For parallel or comparison runs, use neutral filenames and keep subagents unaware of the experimental protocol — double-blind logic. Added as 4th bullet in the CLAUDE.md workflow "Agent architecture" subsection (pending TODO #1 writeout). Surfaced after two concrete poisoning incidents in this session: (a) seeding named authors/works into the first research-dispatch prompt, (b) telling the planned second dispatch it was the "unseeded control."
34. **Meta-pattern: process guards contaminate the thing they try to protect.** Observed three times this session: (a) over-correcting a narrow mistake into a blanket rule = premature bounding with opposite sign; (b) pre-inventing taxonomy to "be safe" = decision-space bounding before it's earned; (c) telling a control-group subagent it's the control = evaluator-awareness contamination. Candidate engram (TODO #15) and candidate pattern-library entry (TODO #17) — not captured as a single doctrine rule yet because each instance lives under its own specific rule in the workflow section, and naming the meta-pattern as a general law would itself be premature bounding (recursive joke partially intended).
35. **Empirical demo of context-poison effect.** Ran the same 16-concept prior-art research three times: (1) seeded with specific author/work hints in the prompt; (2) "clean" prompt but the agent read the first brief (unisolated); (3) fully isolated (no source hints, no references to prior runs, output written to a fresh directory). Comparison showed the seeded run confabulated a multi-source grounding chain for item 16 (creative drilling / open-tabs) — Zeigarnik, Allen, Engelbart, Miller-Galanter-Pribram — that did NOT survive independent search. Isolated run instead concluded the item is a novel synthesis with different component literature (Parking Lot, yak shaving, Tree of Thoughts, Miller 7±2). All three runs converged on items 4, 7, 11 as novel, providing clean triangulation on genuine findings. Conclusion: prompt seeding doesn't just bias selection — it can fabricate grounding dressed as evidence. The SKILL.md "hallucinated citations" anti-pattern applied, but only helped in the isolated run. Implication for future dispatches: isolation is the default; seeding is a pollution source whose cost is paid in false-positive citations that look real.
