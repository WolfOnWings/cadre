## Prose Summary

**Layer 1:** No applicable code surface — the target is a skill direction document, not executable code. L1: no applicable surface.

The single biggest leverage point is decomposing the opaque "draft full plan" beat into three named sub-steps [1]: task-boundary decomposition, schema population, and inline render, because without sub-step names the orchestrator has no anchor for the ≤single-function granularity bar that the direction cites as the accuracy differentiator.

The researcher-cadre selection as the API-grounding reviewer is a primitive-purpose mismatch [2]: researcher-cadre's SOP is brainstorm-space retrieval, not pseudocode-surface verification, and deploying it for API hallucination detection (Spracklen arXiv 2406.10279) without a grounding brief or a purpose-built primitive will yield unchecked hallucination rates in produced plans.

The `traceability` field in the task schema needs a concrete target defined at spec time [3] — without specifying that it maps to a brainstorm-artifact section, the LLM will populate it plausibly but not usefully; and the `acceptance` / `done-criteria` field pair needs a verifier-altitude split [4] to eliminate the overlap a schema reader will otherwise produce.

The two-phase structure (DEVELOP / DELIVER) buries a four-interaction sequence inside DEVELOP [5]: a cleaner three-phase layout (DEVELOP → REVIEW → DELIVER) would give each phase exactly one user-visible output and one gate, matching the book metaphor's sentence/page/chapter coherence criterion.

The parallel reviewer dispatch assumes brooks-review-cadre can fire as an in-context skill inside plan mode [6], which is unverified — plan mode restricts tool execution, and a live-fire test is needed before the SOP commits to that dispatch shape.

The plan-mode soft check is positioned mid-SOP rather than as the SOP's first action [7], which means the skill begins executing before detecting off-mode state; moving the check to the literal first step is the error-prevention > error-recovery application here.

The persist-to-`.cadre/plans/` step [8] repeats ADR-062's known fragility (orchestrator-side SOP step at the highest-compaction moment); the CLOSE-step workaround (copy before ExitPlanMode) is the low-effort guard until the hook infrastructure lands.

The signoff #1 gate label gives the user no anticipation signal about the parallel dispatch it triggers [9] — adding a one-line consequence note before the gate is the Tognazzini anticipation lever; and ExitPlanMode needs a one-line mode-shift signifier [10] so the user knows the session has transitioned from proposal to execution mode (Norman signifier; prevents the failure mode of continued proposal-mode assumptions after the boundary).
