# Optimization Plan — handoff-mx-cadre startup integration

**Target:** `.claude/agents/handoff-mx-cadre.md` + supporting hooks and logger
**Pain:** ~7 min wall-clock for 187-event integration run; 31 tool calls, 22 substantive steps
**Date:** 2026-04-27

---

## Prose Summary

The dominant cost is turn count, not computation — each Bash call costs a full LLM inference step, and the six-call Phase 3 analysis sweep (steps 9–14) alone accounts for roughly 3–4 minutes of the 7-minute total [1]. Collapsing that sweep into a single pre-baked script invocation is the highest-leverage single change. Moving event filtering to logger write-time [2] attacks the same problem from the input side: fewer lines in the file means shorter grep/wc round-trips and a simpler slice calculation, compounding with [1]. Pre-staging the processing rename inside the prime hook [3] eliminates three acquisition round-trips (steps 1–3) that currently precede every run; the hook already owns the sentinel-write and knows the path. The Stop-hook architecture question [4] is the highest-stakes option: shifting integration to session-end makes SessionStart near-instant for the user at the cost of losing integration on crash; this is an architectural trade-off, not a pure optimization, and sits above the others in impact but requires a deliberate user decision. The user's seed suggestion — a pre-baked script file [1] — ranks highest among the non-architectural moves and should be applied regardless of which other options land. Layer 1 surfaces one additional finding: the slice arithmetic (grep sentinel → compute line N → head/tail) is a three-call sequence that a single Python or Bun script can execute as one pass [5]. Layer 4 surfaces one UX gap: the prime hook PRIME_INSTRUCTION tells the orchestrator to read `.cadre/handoff.md` after the subagent returns, but this step fires only if the orchestrator follows the instruction — there is no enforcement; a SessionStart hook that writes the active entry into `additionalContext` would close the loop [6].

## Per-Layer Confidence Calibration

**L1 (Performance) — High.** All four L1 findings are grounded in the action report step count, the events.log line count (187 observed), and the logger source code. The causal chain from turn-count → wall-clock is direct: each additional Bash call costs one LLM inference round-trip plus subprocess overhead. Estimates (6 calls → 1 call; 187 lines → 70–90 lines) are conservative and measurable.

**L2 (LLM Capability Surface) — not applicable.** The synthesizer's core work — narrative compression, signal/noise filtering, arc synthesis — is correctly assigned to LLM judgment. No underuse or over-delegation finding. The Phase 3 inline Python scripts are a harness inefficiency (L1), not an LLM capability misuse.

**L3 (Harness Architecture) — Medium.** Finding [4] (Stop-hook shift) and finding [3] (prime hook pre-staging) are architectural. [3] is clean and low-risk. [4] is higher confidence on the user-experience argument (SessionStart latency is the clearest pain) but medium confidence on the crash-recovery trade-off — the current crash-recovery path in the SOP (`.processing-*` file detection) relies on the rename happening before any failure; Stop-hook integration shifts the failure mode to "session ends without Stop firing," which has different (and potentially worse) recovery semantics that need explicit design.

**L4 (Interaction Design) — High.** Finding [6] is directly grounded in: (a) the user's own statement ("I shouldn't have to tell you"), (b) the `PRIME_INSTRUCTION` text in `.claude/hooks/handoff-mx-prime-cadre.ts` line 24 which relies on text-instruction compliance (L3 trust) rather than hook-injection (L1 trust), and (c) Amershi guideline G8 ("support efficient invocation") — injecting context via hook output is a stronger affordance than an instruction the orchestrator must choose to follow.
