## Pre-Mortem Review Verdict

**Verdict:** revise
**Confidence:** medium

### Failure narratives (top 5, ranked by likelihood)

1. **Knip entry-glob miss leaves skills directory uncovered, producing false-positive dead-code failures on every new skill file.** The plan's `knip.json` defines entry globs for `.claude/hooks/**/*.ts`, `.claude/agents/**/*.ts`, and `scripts/**/*.ts`. Cadre's active growth vector includes `.claude/skills/` — two skill directories (`develop-deliver-cadre/`, `discover-define-cadre/`) are already present in the working tree at implementation time. Any `.ts` file in `.claude/skills/` that exports a symbol has no path from the declared entry graph; Knip reports those exports as unused; the `dead-code` CI job fails on the first skill file committed after the CI lands; the user adds an `ignoreDependencies` or `ignoreBinaries` exception rather than expanding the entry glob; within three iterations the `knip.json` accumulates ignore-list entries that suppress real dead-code signals; the dead-code check becomes a noise source; the user begins ignoring it. Triggering assumption: `.claude/skills/**/*.ts` is not a current or near-future source of TypeScript entry points.

2. **Oxlint silently drops borrowed Lipstyk rules when rule names diverge from the stable namespace.** The plan's `.oxlintrc.json` uses rule names like `"no-explicit-any"`, `"no-magic-numbers"`, `"require-await"`. Oxlint adopted a plugin-namespace schema for ESLint-compat rules (e.g., `eslint/no-magic-numbers`, `typescript/no-explicit-any`) in its stabilization cycle; the bare names without namespace prefix are either silently ignored or aliased inconsistently depending on the oxlint version installed. The plan documents the mapping only in the commit message (concern A denial: no machine-checked sync). `bun install` upgrades oxlint on the next `bun update` run; the namespaced rules stop matching; the lint job passes on code that introduces `any`-typed parameters; the Lipstyk taxonomy is no longer enforced; the discrepancy is discovered when a manual review catches a violation that CI approved. Triggering assumption: oxlint's bare rule names remain stable across minor version upgrades without namespace prefix qualification.

3. **`check-test-exists` exits 1 on Day 1, aggregator fails, and TODO #23 branch protection cannot be enabled during the highest-velocity window.** The plan explicitly documents "exit 1 today (intentional)" and "gate is failing today by design." The downstream TODO #23 configures branch protection to require `ci-complete-bootstrap`. If TODO #23 is enabled while `test-exists` is failing, every PR is unconditionally blocked — including the PR that would add the first test. The user defers enabling TODO #23 until the test exists; the test does not exist because there is no gating pressure to write one; the "forcing function" operates in reverse (the CI that was meant to force test-writing is itself blocked from being a gate by the absence of tests); the CI foundation sits installed but not enforcing for weeks. Triggering assumption: the user will write the first test before enabling TODO #23 branch protection, independently of any CI gate.

4. **comment-ratio baseline check fails on existing hook or agent files at Task 8, stalling the implementing session before the smoke test.** The plan's `check-comment-ratio.ts` scopes to `.claude/hooks/**/*.ts`, `.claude/agents/**/*.ts`, and `scripts/**/*.ts`. The "spot check" validating the ≥10% threshold covers only `handoff-mx-logger-cadre.ts` (~30%). Other files in `.claude/hooks/` or `.claude/agents/` may have lower ratios — particularly generated or stub files. If 2–3 files fail the 10% threshold on the first run of Task 8, the implementing agent faces an undocumented branch: lower the threshold (defeats the gate), retroactively comment existing files (scope creep), or mark files exempt. None of these branches appear in the plan. The implementing session stalls; the agent escalates to the user; the smoke test at Task 11 is not reached in the session; the CI PR is never opened during that session. Triggering assumption: all currently existing files in `.claude/hooks/` and `.claude/agents/` have ≥10% comment ratio — validated by spot-checking a single file.

5. **`.tsbuildinfo` incremental cache is never persisted in CI, making every typecheck job cold despite `incremental: true`.** The plan adds `.tsbuildinfo` to `.gitignore` (Task 3, correctly) and sets `incremental: true` in `tsconfig.json` (Task 2). The `ci.yml` cache key is `hashFiles('bun.lock')` (Task 10) — keyed to the lockfile, not the TS source. `.tsbuildinfo` is neither committed nor included in the Actions cache configuration. Every CI run triggers a full cold `tsc --noEmit`; the incremental cache provides zero benefit; as the TS file count grows the typecheck job becomes the slowest step; the "fast L1 floor" intent erodes; the user adds a wall-clock complaint that the plan explicitly dropped from success criteria. Triggering assumption: the `hashFiles('bun.lock')` cache entry is sufficient to persist the tsc incremental output between CI runs.

### Top concerns

1. **Knip entry-glob miss** — `.claude/skills/**/*.ts` is already populated and growing; it is not in the `knip.json` entry globs; the dead-code job will produce false positives on every new skill file from Day 1 of real usage. The fix is a one-line addition to `knip.json` entry, but the plan as written will produce noise immediately.

2. **Oxlint rule-name namespace drift** — the commit-message-only mapping (concern A denial) leaves no machine-checkable anchor; oxlint's bare names versus namespaced names is a known stabilization-era ambiguity; the lint gate can appear green while the Lipstyk taxonomy is silently not enforced. The fix requires qualifying rule names with their namespace prefix in `.oxlintrc.json`.

3. **test-exists / TODO #23 ordering deadlock** — the plan treats the `exit 1 today` failure as acceptable but does not document the ordering dependency with TODO #23. Without an explicit note that branch protection must not be required until the first test lands, the implementing session leaves an unresolved dependency that will cause confusion or CI paralysis in the next session.

### Steelman of alternative *(only if verdict != proceed)*

A staged CI approach — typecheck + lint only in v1, dead-code and structural checks in v2 — would eliminate narratives 1 and 4 entirely: Knip entry-glob coverage and comment-ratio baseline validation are complexity that requires stabilized scope to configure correctly. The v1 gate would be unambiguously green (or red on real type/lint violations), branch protection could be enabled immediately, and the entry-glob and threshold questions could be answered with real data from a running codebase. The tradeoff is that the "throw the book" framing is deferred, and the user explicitly chose the wide-net approach. The staged alternative is the right steelman if narratives 1 or 4 cause session stalls — not a direction change, but a sequencing revision that keeps the CI PR shippable without the stub-and-fix branches the current plan leaves unresolved.
